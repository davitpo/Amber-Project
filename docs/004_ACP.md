# Amber Command Protocol

Version 1

## Philosophy
*   **Commands** are written for humans: `COMMAND [arguments...]` (space-separated, whitespace tolerant, case-insensitive).
*   **Responses** are written for software: `KEY=VALUE` pairs (ensuring parsers, Android Apps, LabVIEW or python utilities read machine-specific structures clearly).

---

## Amber Bluetooth Low Energy (BLE) Service
Amber publishes its own proprietary bidirectional service schema profile:

### UUIDs
*   **Amber Service UUID**: `12345678-1234-5678-1234-56789abcdef0`
*   **RX Characteristic UUID** (Properties: `WRITE`, `WRITE WITHOUT RESPONSE`): `12345678-1234-5678-1234-56789abcdef1`
*   **TX Characteristic UUID** (Properties: `NOTIFY`, `READ` plus `0x2902` Client Characteristic Configuration Descriptor): `12345678-1234-5678-1234-56789abcdef2`

### Notification Subscription
The connected client (e.g. mobile companion app) must explicitly subscribe to notifications on TX (`12345678-1234-5678-1234-56789abcdef2`) prior to exchanging ACP packets.

---

## BLE Notification Chunking & Queue
*   **Bounded static TX Queue**: A non-blocking Ring-Buffer allocates static memory array chunks on start, bypassing heap allocations.
*   **Chunking (20-byte payload limit)**: Response packets exceeding safe bounds are split into 20-byte notifications sequentially (with 15ms stack delays inside loop checks) preventing system stalls.

---

## Command Syntax
Whitespace characters act as delimiters between the target command keyword and the corresponding arguments parameter.
*   **Legacy equals-sign delimiters are strictly rejected** with `ERROR=INVALID_SYNTAX` to ensure parse uniformity.

```bash
# Set local clock value
SETTIME 12:36:45

# Set dimming brightness percent
BRIGHTNESS 80
```

---

## Command: GETTIME
Reads current local clock time. 

Note: The returned value represents Amber's current local clock software model time, and does not yet imply real-time clock (RTC), NTP, or automatic mobile phone time synchronization.

### Syntax
`GETTIME` (Arguments: 0. Exceeding parameters return `ERROR=INVALID_SYNTAX`).

### Response
`TIME=hh:mm:ss` (Zero-padded, 24h format).

#### Example:
```bash
ACP(UART) > GETTIME
ACP(UART) < TIME=12:36:45
```

---

## Command: SETTIME
Sets Amber's internal clock runtime values. 

### Syntax
`SETTIME hh:mm:ss` (Exactly 1 space is expected before parameter. hh: 0..23, mm: 0..59, ss: 0..59).

### Response
`OK`

#### Examples:
```bash
ACP(UART) > SETTIME 12:36:00
ACP(UART) < OK

# Out of bounds value validation rejection
ACP(UART) > SETTIME 25:00:00
ACP(UART) < ERROR=OUT_OF_RANGE

# Incorrect syntax formatting validation rejection
ACP(UART) > SETTIME now
ACP(UART) < ERROR=INVALID_FORMAT
```

---

## Command: GETBRIGHTNESS
Exposes current display brightness percentage value.

### Syntax
`GETBRIGHTNESS` (Arguments: 0. Exceeding parameters return `ERROR=INVALID_SYNTAX`).

### Response
`BRIGHTNESS=<0..100>`

#### Examples:
```bash
ACP(UART) > GETBRIGHTNESS
ACP(UART) < BRIGHTNESS=100
```

---

## Command: BRIGHTNESS
Sets local display backlight brightness level asynchronously using on-board driver PWM.

Note: Input values correspond to perceptual/user-facing levels (percentage). The raw hardware PWM parameters and exponential gamma lookup curves are entirely abstract and evaluated internally inside the `Display` block.

### Syntax
`BRIGHTNESS <0..100>` (Value range strictly checked within bounds `0` and `100`).

### Response
`OK`

#### Examples:
```bash
ACP(UART) > BRIGHTNESS 36
ACP(UART) < OK

# Out of bounds rejection
ACP(UART) > BRIGHTNESS 105
ACP(UART) < ERROR=OUT_OF_RANGE

# Syntax mismatch rejection
ACP(UART) > BRIGHTNESS high
ACP(UART) < ERROR=INVALID_FORMAT
```

## Console Direction Markers
ACP requests and responses are flagged visually with direction headers:
*   `>` Request indicator (e.g. `ACP(UART) > PING`)
*   `<` Response indicator (e.g. `ACP(UART) < PONG`)

### Multiline Output
Multiline answers output the direction header followed by the target lines, terminating on `END`:

```bash
ACP(UART) > STATUS
ACP(UART) <
TIME=00:36:00
RENDER_MODE=FullSprite
BLE=DISCONNECTED
HEAP_FREE=124000
HEAP_MIN=119000
FPS=30
UPTIME_MS=10500
END
```

## BLE Life-Cycle Event Logs
Core BLE hardware states publish standard prefixing headers:
*   `BLE EVENT: Initializing`
*   `BLE EVENT: Service created`
*   `BLE EVENT: RX ready`
*   `BLE EVENT: Advertising started`
*   `BLE EVENT: Connected`
*   `BLE EVENT: Disconnected`
*   `BLE EVENT: Advertising restarted`

---

## Memory Diagnostics Checkpoints on Startup
Available RAM bounds are tracked at critical startup transitions to calculate accurate component consumption:
*   `MEM BOOT` Startup state memory.
*   `MEM DISPLAY` Backlight and physical driver initializations impact.
*   `MEM SPRITE` Double buffer canvas allocation impact.
*   `MEM BLE_BEFORE` Heap right before BLE startup.
*   `MEM BLE_AFTER` Heap right after BLE startup.
*   `MEM BLE_COST` Exact cost of the BLE stack (calculated exclusively as `MEM BLE_BEFORE - MEM BLE_AFTER`).

## Machine-Oriented Info Query (`INFO`)
The machine-centric `INFO` operation queries operational metadata directly from the compile-time `CommandRegistry`:

#### Structure:
`INFO [CommandName]` or `INFO ALL`

#### Output Trace Example:
```bash
COMMAND=SETTIME
CATEGORY=CLOCK
STATUS=PLANNED
ARGS=1
ARG1=Target clock time
FORMAT=hh:mm:ss
```

Help messages intended for humans output cleanly formatted, visually aligned tables:

```bash
Command     : SETTIME
Category    : Clock
Usage       : SETTIME <hh:mm:ss>
Description : Set local clock.
Status      : Planned

```

## Android Client Integration Checklist
1. **Discovery & Connection**: Scan for advertise device profile `Amber Clock`. Connect to physical MAC address.
2. **Handle Services**: Discover Service `12345678-1234-5678-1234-56789abcdef0`.
3. **Register Notifications**: Subscribe to notifications on TX Characteristic `12345678-1234-5678-1234-56789abcdef2` (ensure client descriptors write `0x0001` to CCD descriptor `0x2902` configuration handles).
4. **Command Exchange**: Write standard Command to Characteristic RX `12345678-1234-5678-1234-56789abcdef1`.
5. **Collate responses stream**: Assemble separate 20-byte notifications received asynchronously until checking terminal responses (such as `PONG`, `OK`, `ERROR=...` or multiline ending sequence marker text `END`).
