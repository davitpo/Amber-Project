# Amber Command Protocol

Version 1

## Philosophy
*   **Commands** are written for humans: `COMMAND [arguments...]` (space-separated, whitespace tolerant, case-insensitive).
*   **Responses** are written for software: `KEY=VALUE` pairs (ensuring parsers, Android Apps, LabVIEW or python utilities read machine-specific structures clearly).

---

## Command Syntax
Whitespace characters act as delimiters between the target command keyword and the corresponding arguments parameter.
*   **Legacy equals-sign delimiters are strictly rejected** with `ERROR=INVALID_SYNTAX` to ensure parse uniformity.

```bash
# Set local clock value
SETTIME 12:36:45

# Set dimming brightness percent
BRIGHTNESS 80
```

---

## Command: GETTIME
Reads current local clock time. 

Note: The returned value represents Amber's current local clock software model time, and does not yet imply real-time clock (RTC), NTP, or automatic mobile phone time synchronization.

### Syntax
`GETTIME` (Arguments: 0. Exceeding parameters return `ERROR=INVALID_SYNTAX`).

### Response
`TIME=hh:mm:ss` (Zero-padded, 24h format).

#### Example:
```bash
ACP(UART) > GETTIME
ACP(UART) < TIME=12:36:45
```

---

## Console Direction Markers
ACP requests and responses are flagged visually with direction headers:
*   `>` Request indicator (e.g. `ACP(UART) > PING`)
*   `<` Response indicator (e.g. `ACP(UART) < PONG`)

### Multiline Output
Multiline answers output the direction header followed by the target lines, terminating on `END`:

```bash
ACP(UART) > STATUS
ACP(UART) <
TIME=00:36:00
RENDER_MODE=FullSprite
BLE=DISCONNECTED
HEAP_FREE=124000
HEAP_MIN=119000
FPS=30
UPTIME_MS=10500
END
```

## BLE Life-Cycle Event Logs
Core BLE hardware states publish standard prefixing headers:
*   `BLE EVENT: Initializing`
*   `BLE EVENT: Service created`
*   `BLE EVENT: RX ready`
*   `BLE EVENT: Advertising started`
*   `BLE EVENT: Connected`
*   `BLE EVENT: Disconnected`
*   `BLE EVENT: Advertising restarted`

---

## Memory Diagnostics Checkpoints on Startup
Available RAM bounds are tracked at critical startup transitions to calculate accurate component consumption:
*   `MEM BOOT` Startup state memory.
*   `MEM DISPLAY` Backlight and physical driver initializations impact.
*   `MEM SPRITE` Double buffer canvas allocation impact.
*   `MEM BLE_BEFORE` Heap right before BLE startup.
*   `MEM BLE_AFTER` Heap right after BLE startup.
*   `MEM BLE_COST` Exact cost of the BLE stack (calculated exclusively as `MEM BLE_BEFORE - MEM BLE_AFTER`).

## Machine-Oriented Info Query (`INFO`)
The machine-centric `INFO` operation queries operational metadata directly from the compile-time `CommandRegistry`:

#### Structure:
`INFO [CommandName]` or `INFO ALL`

#### Output Trace Example:
```bash
COMMAND=SETTIME
CATEGORY=CLOCK
STATUS=PLANNED
ARGS=1
ARG1=Target clock time
FORMAT=hh:mm:ss
```

## Human-Friendly Help Output (`HELP`)
Help messages intended for humans output cleanly formatted, visually aligned tables:

```bash
--------------------------------
Command     : SETTIME
Category    : Clock
Usage       : SETTIME <hh:mm:ss>
Description : Set local clock.
Status      : Planned
--------------------------------
```

## Architecture & Transport Independence
The Amber Command Protocol (ACP) v1 is designed as a transport-independent ASCII communication framework. While initially integrated via Bluetooth Low Energy (BLE), the parser layer is fully isolated from hardware boundaries, facilitating future expansions to other channels such as:
*   **UART / Serial** (e.g., direct USB diagnostics)
*   **USB** (native virtual COM port endpoints)
*   **TCP/IP** (over future cellular/Wi-Fi socket adapters)

Command sources are labeled dynamically using `CommandSource` headers (e.g., `ACP(BLE)` or `ACP(UART)` prefixes).

---

## Serial / UART Framing
All commands received through UART must terminate with a line-ending indicator:
*   Line Feed (`\n`)
*   Carriage Return (`\r`)
*   Combination carriage-return line feed (`CRLF`)

Buffer arrays assemble complete messages asynchronously without halting standard application updates.

### Buffer Bounds
*   **64 Bytes Limit**: Maximum logical command storage size. 
*   **Oversized Line Recovery**: Lines in excess of 64 bytes immediately return an `ERROR=TOO_LONG` and drop any trailing stream characters until a valid line ending resets the incoming buffer, ensuring reliable subsequent processing.

---

## Command Registry
ACP utilizes a single source of truth database (`CommandRegistry`). Each operational command lists corresponding categories, parameter syntax descriptions, and current lifecycle configuration states. 

The command database remains static and is allocated with zero dynamic overhead.

### Initial Categories:
- **System**: Base operations (`PING`, `VERSION`, `HELP`)
- **Clock**: Temporal queries (`GETTIME`, `SETTIME`)
- **Display**: Screen properties (`GETBRIGHTNESS`, `BRIGHTNESS`)
- **Diagnostics**: Health reports (`STATUS`)

---

## Dynamic HELP Database
Rather than relying on static hardcoded strings, help systems generate reference listings directly from the registry:

### Grouped Commands (`HELP`)
Retrieves the complete catalog categorized dynamically:
`ACP Version 1` followed by categorized listing titles.

### Focused Helper (`HELP=<COMMAND>`)
Query parameters explicitly, printing:
- `Command`
- `Syntax`
- `Description`
- `Status`
